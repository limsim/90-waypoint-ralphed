  Use plain `rm -rf dist`.
- Layering enforced by project-reference tsconfigs: `core` (domain+application, NO DOM lib) <- `adapters`
  (+DOM, includes src/main.ts) and `test` (+node types). Relative imports need a `.js` extension.
- `SeededRandom` (PRNG implementation) lives in `src/domain/` not adapters — `tsconfig.test.json` only references core (domain+application), so tests can't import from adapters. Any pure implementation needed in tests must be in domain/ or application/.
- Expanding `RandomSource` interface: search for `{ nextFloat:` stubs across tests/ whenever adding a new method — each stub must implement the full interface or tsc fails.
- layout-rules adjacency model: `segments[i]` (0-indexed) is adjacent to `waypoints[i]` and
  `waypoints[i+1]`. Everything else is "non-adjacent". US-008 (Walk) and US-010 (generator) must use
  this same index convention when calling the layout-rule predicates.
- `Walk.create(waypoints, bounds)` (src/domain/walk.ts) DERIVES segments from consecutive waypoints
  (`new Segment(wps[i].position, wps[i+1].position)`) — callers (US-010 generator) pass ONLY the placed
  waypoints + the generation region, never a separate segments array. This makes the layout-rules
  adjacency model true by construction. `bounds` = generation REGION (for the within-bounds 30px check);
  `walk.boundingBox` = the separate TIGHT box around waypoints (US-013/US-015 render/A4-fit use this one).
- layout-rules predicates take mutable `Waypoint[]`/`Segment[]`; when holding `ReadonlyArray`, spread-copy
  (`[...waypoints]`) before calling `checkLayout`/predicates or tsc rejects the readonly→mutable arg.
- `layout-rules.ts` EXPORTS `turnLabelPoint(position: Point): Point` (NE 45°, 46px). The walk-generator
  reuses it for its incremental label-clearance check so the hot loop and the `Walk` invariant share the
  exact same label geometry (no divergence). Reuse it in US-014 (renderer) too rather than recomputing.
- The walk-generator (`src/domain/walk-generator.ts`) places relative coords then CENTRES them in the
  bounds: spacing constraints are translation-invariant, so only the within-bounds rule cares about
  absolute position, and after centring that rule is just a size check (`bboxW+60 <= bounds.width`).
  Growth fixes size; whole-sequence re-roll fixes spacing (ADR-0002). GREEDY placement dead-ends badly
  for large counts (~90) — placement MUST use bounded backtracking over segment-length multipliers.
- A `function*` defers its body until the first `.next()`, so an argument guard like `if (count<2) throw`
  inside the generator fires on the first `.next()`, NOT when `generate(...)` is called. Tests that drive
  the iterator (calling `.next()`) still see the throw; a bare `generate(1, rng)` would not.
- walk-generator's count guard is `!Number.isInteger(count) || count < 2` — a fractional count would
  otherwise throw an opaque `RangeError: Invalid array length` deep in placement (`new Array(2.5)`).
  Validate finite-integer-ness at any domain boundary that feeds `new Array(n)`.
- To test the walk-generator's bounds-growth clause IN ISOLATION (not just the failure path), use a
  small `initialRegion` no placement fits and A/B the SAME seed on `maxGrowths`: a too-low value fails,
  the default succeeds. Placement is bounds-independent, so `maxGrowths` is the only variable. (Seed
  4242, count=10, 200x200 region needs exactly 4 growths.) When refining an already-correct story,
  prefer adding tests for ACs that are implemented-but-never-asserted — grep existing tests per AC
  phrase first (e.g. wildcard COUNT was tested but wildcard POSITION-variance was not).
- To exercise the walk-generator's per-batch (~50 attempts) progress yield deterministically, reuse
  the impossible-fit setup (100x100 initialRegion, maxGrowths:0, count=10 — nothing fits, proven by
  the failure test) but raise maxPlacementAttempts ABOVE yieldEvery (e.g. 120): re-roll 0 then runs
  the full inner loop and the `attempts % yieldEvery === 0` yields fire at 50/100 with rerolls still
  0. Same setup pins the failure signal's running totals (attempts === maxPlacementAttempts*sizes,
  rerolls === maxRerolls). It's robust because the inner loop always runs to its cap regardless of
  whether any individual placeOnce succeeds (attempts++ happens before placeOnce is called).
- Application use cases live in `src/application/` and depend ONLY on domain + ports (tsconfig.core
  has no DOM lib, so a stray DOM reference fails to compile — the layering is mechanically enforced).
  `GenerateWalk` (src/application/generate-walk.ts) drives the synchronous walk-generator iterator
  with `let s = it.next(); while(!s.done){ await port.yieldToEventLoop(); s = it.next(); } return s.value;`
  — DISCARD the yielded progress; the terminal `s.value` is the GenerationResult, returned verbatim.
- `yield` is a RESERVED word in ESM/strict scope — you cannot name a local/param/field `yield`. Name
  Yield-port fields/vars `eventLoopYield` (or similar). (A property access like `obj.yield` is legal,
  but a bare identifier is not — don't risk it.)
- Inject STABLE collaborators at construction (the Yield port) but pass PER-GENERATION state as a
  method arg (the RandomSource on `execute(count, random, config?)`). A shared mutable SeededRandom
  reused across generations would make walk N depend on all prior draws, breaking US-022's
  single-seed reproducibility — so each Generate must get a FRESH seeded source. The use case stays
  seed-agnostic; seed choice/URL handling is the adapter/composition layer's job (US-021/US-022).
- Test an async use case that wraps a synchronous generator with two Yield doubles: an immediate
  no-op (`{ yieldToEventLoop: () => Promise.resolve() }`) for behaviour assertions, and a CountingYield
  (increments a `calls` field) to prove the event loop is actually freed (>=1 on success, >=1 per
  re-roll on the failure path). The strongest correctness check is a "faithful driver" test: assert
  the use-case result equals the result of driving `walkGenerator.generate` synchronously for the
  SAME seed — the async wrapper must not perturb the random stream. A `function*` count guard fires
  on the first `.next()`, so `execute(badCount, ...)` REJECTS — assert via `assert.rejects`.
