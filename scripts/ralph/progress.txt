- layout-rules adjacency model: `segments[i]` (0-indexed) is adjacent to `waypoints[i]` and
  `waypoints[i+1]`. Everything else is "non-adjacent". US-008 (Walk) and US-010 (generator) must use
  this same index convention when calling the layout-rule predicates.
- `Walk.create(waypoints, bounds)` (src/domain/walk.ts) DERIVES segments from consecutive waypoints
  (`new Segment(wps[i].position, wps[i+1].position)`) — callers (US-010 generator) pass ONLY the placed
  waypoints + the generation region, never a separate segments array. This makes the layout-rules
  adjacency model true by construction. `bounds` = generation REGION (for the within-bounds 30px check);
  `walk.boundingBox` = the separate TIGHT box around waypoints (US-013/US-015 render/A4-fit use this one).
- layout-rules predicates take mutable `Waypoint[]`/`Segment[]`; when holding `ReadonlyArray`, spread-copy
  (`[...waypoints]`) before calling `checkLayout`/predicates or tsc rejects the readonly→mutable arg.
- `layout-rules.ts` EXPORTS `turnLabelPoint(position: Point): Point` (NE 45°, 46px). The walk-generator
  reuses it for its incremental label-clearance check so the hot loop and the `Walk` invariant share the
  exact same label geometry (no divergence). Reuse it in US-014 (renderer) too rather than recomputing.
- The walk-generator (`src/domain/walk-generator.ts`) places relative coords then CENTRES them in the
  bounds: spacing constraints are translation-invariant, so only the within-bounds rule cares about
  absolute position, and after centring that rule is just a size check (`bboxW+60 <= bounds.width`).
  Growth fixes size; whole-sequence re-roll fixes spacing (ADR-0002). GREEDY placement dead-ends badly
  for large counts (~90) — placement MUST use bounded backtracking over segment-length multipliers.
- A `function*` defers its body until the first `.next()`, so an argument guard like `if (count<2) throw`
  inside the generator fires on the first `.next()`, NOT when `generate(...)` is called. Tests that drive
  the iterator (calling `.next()`) still see the throw; a bare `generate(1, rng)` would not.
- walk-generator's count guard is `!Number.isInteger(count) || count < 2` — a fractional count would
  otherwise throw an opaque `RangeError: Invalid array length` deep in placement (`new Array(2.5)`).
  Validate finite-integer-ness at any domain boundary that feeds `new Array(n)`.

- To exercise the walk-generator's per-batch (~50 attempts) progress yield deterministically, reuse
  the impossible-fit setup (100x100 initialRegion, maxGrowths:0, count=10 — nothing fits, proven by
  the failure test) but raise maxPlacementAttempts ABOVE yieldEvery (e.g. 120): re-roll 0 then runs

  iterator synchronously then await once at the end" and "don't await at all" — both of which the
  count-based tests pass straight through (a single end-yield still counts as >= 1, and a sync drain
  still produces the right result object). With the new test, either makes the external macrotask fire

## 2026-06-28 - US-017 (Waypoint click tooltip and hover highlighting) — REFINEMENT
- US-017 already passed; re-verified all acceptance criteria green end to end (npm test 243 pass;
  verify:renderer all assertions; verify:controls 22/22; typecheck clean) before refining.
- Refinement (impl + harness): made `CanvasRenderer.hitTest` BORDER-AWARE — a genuine AC3
  ("hit-testing inverts both transforms: viewport → A4 → generation") correctness fix.
  - The bug: `getBoundingClientRect()` returns the element's BORDER box, but a canvas's backing store
    maps to its CONTENT box, and index.html gives `#walk-canvas` a 1px border. The old inversion used
    `(client - rect.left) / rect.width * canvas.width`, treating the border box AS the content box —
    so the viewport→backing mapping was skewed off-centre. Negligible at full size with a 1px border
    (sub-pixel), but the error GROWS as the canvas element is CSS-downscaled on a narrow viewport
    (US-015's `canvas { max-width: 100% }`), because the border does NOT scale with the content.
  - The fix: subtract the (uniform) border from BOTH the origin and the size before scaling —
    `borderX = canvas.clientLeft; contentW = rect.width - 2*borderX; backing = (client - rect.left -
    borderX) / contentW * canvas.width` (same for Y); guard `contentW/contentH <= 0` (replaces the old
    `rect.width === 0` guard). `clientLeft/clientTop` are 0 on an unbordered canvas, so this reduces to
    the prior border-box mapping (no behavioural change where there is no border).
- Files changed:
  - src/adapters/canvas-renderer.ts (hitTest border-box → content-box correction + doc comment)
  - scripts/verify-renderer.mjs (verifyHitTest gained an optional `border` param: the fake canvas now
    models a border box with matching clientLeft/clientTop/clientWidth/clientHeight, and the client
    coords are derived through the CONTENT box; added a `border 24` + half-size count=90 call site)
  - src/adapters/CLAUDE.md (documented the border correction under the hitTest section)
  - scripts/ralph/progress.txt (this entry + a Codebase Patterns bullet)
- PROVEN TO BITE then reverted via precise Edit (NOT git checkout — files uncommitted): reverting
  hitTest to `(client - rect.left) / rect.width` failed the new `border 24` case at wp1 (an outer
  waypoint no longer within WAYPOINT_RADIUS of its true centre) while every `border 0` case still
  passed; restored + `npm run clean` + re-ran all gates green.
- **Learnings for future iterations:**
  - `getBoundingClientRect()` is the BORDER box; a canvas backing store maps to the CONTENT box. Any
    screen→canvas inversion must remove the element border (`clientLeft`/`clientTop`) from the origin

- **Learnings for future iterations:**
  - `print-color-adjust: economy` (the browser default) DROPS CSS `background` colours/images in print;
    BORDERS print regardless. Any printed UI relying on a `background:` fill (here the black Start/End
    legend swatch) needs `print-color-adjust: exact` (+ `-webkit-` prefix) or it prints empty. A
    `<canvas>` bitmap is foreground content and prints regardless — this only bites CSS backgrounds.

    ⊂ `-webkit-print-color-adjust`), use a `(?<!-)` lookbehind to require the unprefixed/spec property,
    so the prefix alone can't satisfy the check vacuously.

    (here: what browsers do to backgrounds at print time), not just adding more structural regex checks
    — the existing gate proved the rules were PRESENT but said nothing about whether they'd print right.
